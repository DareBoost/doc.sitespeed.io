Sitespeed.io 1.7 release notes

Output as JUnit XML

Sitespeed.io can now output JUnit XML that you can use in your continuous integration tool. With that, you can for every build test your site against the rules! In the documentation you will see how to do it for Jenkins & Travis (Travis don't have full support yet for JUnit XML but you can still make sure that builds get broken if you violate the rules).

You can specify your own limits of when you want the tests to fail: for the overall score and the limit individual tests. And you can of course skip tests. Using Jenkins, you will get information per page, and per test, so you easy can see what is failing.

Feed sitespeed.io with a list of URL:s instead of a crawl

For some sites, crawling the page isn't the best thing, you want to test a couple of the most used pages (some of them longer down in the path tree), so now you can supply a list of url:s to sitespeed to test.

The list is a plain text file with one url on each line. The first url will gave the name of the whole test (= the same as the first url to crawl). The file needs to look like this: 

http://peterhedenskog.com/
http://peterhedenskog.com/work/
http://peterhedenskog.com/music/

Check the doc of how to do it. Heads up: The parameter to supply a file is f meaning that the old functionality "follow only the oath when crawling" has changed to parameter c so please make sure you change that if you use that functionality.

More fixes

  • DOM elements

    Added average & median number of DOM elements on site summary and specific number on page.

  • Missing expire

    Show the amount of assets that don't have an exipre header, both on summary and pages summary.

  • Cacheable assets

    Show the percentage of assets that are cacheable.

  • New page summary

    See cache, content, time, content-type and domain section.

Small fixes

  • Page summary: Added new max size of a document for getting a warning, now using stats from HTTPArchive
  • Page summary & site summary: Removed prime cache values, they wasn't actually always right
  • Removed Yslow rule ynumreq and created three new ones in order to get clearer JUnit xml result (and also only check for sync js): cssnumreq, cssimagesnumreq & jsnumreq
  • Added response headers info on assets page (missing in last release)

Bug fix

  • If a max age HTTP cache header was missing the cache time, the cache time was set to 0 instead of checking if a Expire header exists.
  • If an asset was missing expire headers, it inherited the last one assets with an expire date
